# Components, Layouts, and Runtime Behavior

Use this reference when wiring component maps, layouts, providers, `providerImportSource`, `useMDXComponents`, or Next `mdx-components.tsx`.

Source-condensed from this repository's docs:
`docs/docs/using-mdx.mdx`, `docs/guides/injecting-components.mdx`, `docs/table-of-components.mdx`, and `packages/react/readme.md`.

## Compiled Output Model

An MDX file compiles to a module whose default export is an `MDXContent` component. Named ESM exports remain named exports.

```mdx
export function Thing() {
  return <>World</>
}

# Hello <Thing />
```

This roughly becomes:

```jsx
export function Thing() {
  return <>World</>
}

export default function MDXContent(props = {}) {
  return <h1>Hello <Thing /></h1>
}
```

The actual output compiles JSX to runtime calls and reads `props.components`.

## Props

Data can be passed as props:

```mdx
# Hello {props.name.toUpperCase()}

The current year is {props.year}
```

Use:

```jsx
<Post name="Mars" year={2026} />
```

Prefer props for page-specific data. Prefer imports for shared helpers/components.

## Component Maps

The special `components` prop maps MDX component names and markdown-generated element names to implementations:

```jsx
<Post
  components={{
    h1: 'h2',
    em(props) {
      return <i style={{color: 'goldenrod'}} {...props} />
    },
    Planet() {
      return 'Neptune'
    },
    theme: {
      text(props) {
        return <span style={{color: 'grey'}} {...props} />
      }
    }
  }}
/>
```

The map can contain:

- HTML equivalents generated by markdown such as `h1`, `a`, `p`, `pre`, `code`, `ul`, `li`.
- `wrapper`, a special layout key.
- Component names used in JSX such as `Planet`.
- Object members for component expressions such as `<theme.text>`.

## Name Resolution

Rules for `<x>` in JSX:

- If the name has a dot, it is a member expression: `<a.b>` references `a.b`.
- If it is not a valid JS identifier, it is a literal tag: `<a-b>`.
- If it starts lowercase, it is a literal tag: `<a>`.
- Otherwise, it is a component reference: `<A>`.

Important consequence: `components={{a: Link}}` affects markdown links (`[x](url)`) but not literal JSX `<a href="url">x</a>`. Use `<Link>` in MDX or rewrite JSX tags if you want component behavior.

## Precedence

Component precedence:

1. Locally defined or imported components in the MDX file.
2. Components provided by `providerImportSource` or context providers.
3. Components passed through `props.components`.
4. Built-in string element defaults such as `h1: 'h1'`.

In practice, locally imported MDX components override passed components for the same name.

## Layouts

A local layout can be defined as the default export:

```mdx
export default function Layout({children}) {
  return <main>{children}</main>
}

All the things.
```

Or re-exported:

```mdx
export {Layout as default} from './components.js'
```

A layout can also be passed as `components.wrapper`, but a local layout takes precedence.

## Providers

Prefer passing components directly:

```jsx
<Post components={components} />
```

Use a provider when nested MDX imports make prop passing too repetitive:

```jsx
import {MDXProvider} from '@mdx-js/react'

<MDXProvider components={components}>
  <Post />
</MDXProvider>
```

Provider setup requires the MDX compiler integration to use:

```js
{providerImportSource: '@mdx-js/react'}
```

Provider packages are available for React, Preact, and Vue. They are not required for MDX to work.

Nested providers merge component maps by default. To replace instead of merge, pass a function as `components`.

## Component Injection with `providerImportSource`

`providerImportSource` tells the compiled MDX to import `useMDXComponents` from a module.

Create a module:

```jsx
/** @import {MDXComponents} from 'mdx/types.js' */

/** @returns {MDXComponents} */
export function useMDXComponents() {
  return {
    h1(props) {
      return <h2 {...props} />
    },
    Planet() {
      return 'Pluto'
    }
  }
}
```

Configure:

```js
mdx({providerImportSource: import.meta.resolve('./mdx-components.js')})
```

This is the underlying pattern used by MDX providers and Next's `mdx-components.tsx`.

## Next.js Component Injection

In Next.js, add `mdx-components.tsx` in the project root or `src/` and export `useMDXComponents`.

```tsx
import type {MDXComponents} from 'mdx/types'

export function useMDXComponents(components: MDXComponents): MDXComponents {
  return {
    ...components,
    a: (props) => <a {...props} />,
    h2: (props) => <h2 className="section-title" {...props} />
  }
}
```

Do not use `@mdx-js/react` or `providerImportSource` for component injection in Next.js.

## Missing Components

If MDX references `<Alert />` and it is neither imported/defined nor provided, runtime code throws a missing component error. Fix by:

- Importing it in the MDX file.
- Defining it in the MDX file.
- Passing it through `components`.
- Providing it through `providerImportSource` or framework-specific component injection.
